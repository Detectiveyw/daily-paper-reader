## 摘要
云端托管的大语言模型（LLMs）已成为智能体系统（agentic systems）中事实上的规划器，负责协调工具并指导本地环境的执行。然而，在许多部署场景中，被规划的环境是私有的，包含源代码、文件、凭据和元数据，这些信息不能暴露给云端。现有解决方案解决了相关问题，如执行隔离、访问控制或机密推理，但它们并未控制云端规划器在规划过程中观察到的内容：在许可范围内，原始环境状态仍然会被暴露。我们提出了 PlanTwin，一种用于云辅助规划的隐私保护架构，无需暴露原始本地上下文。其核心思想是将真实环境投影到一个“面向规划的数字孪生”（planning-oriented digital twin）中：这是一个受模式约束且去标识化的抽象图，它保留了与规划相关的结构，同时删除了可重构的细节。云端规划器仅通过受限的能力接口在这个经过清理的孪生体上运行，而本地网关（gatekeeper）则强制执行安全策略和累积披露预算。我们进一步将隐私与效用的权衡形式化为能力粒度问题，使用 (k,δ)-匿名性和 ε-不可链接性定义了架构隐私目标，并通过多轮披露控制减轻了组合泄露。我们将 PlanTwin 实现为本地智能体与云端规划器之间的中间件，并在 10 个领域的 60 个智能体任务上使用 4 个云端规划器进行了评估。PlanTwin 实现了完全的敏感项非披露（SND = 1.0），同时保持了接近全上下文系统的规划质量：四个规划器中有三个实现了 PQS > 0.79，且整个流水线的效用损失低于 2.2%。

## Abstract
Cloud-hosted large language models (LLMs) have become the de facto planners in agentic systems, coordinating tools and guiding execution over local environments. In many deployments, however, the environment being planned over is private, containing source code, files, credentials, and metadata that cannot be exposed to the cloud. Existing solutions address adjacent concerns, such as execution isolation, access control, or confidential inference, but they do not control what cloud planners observe during planning: within the permitted scope, \textit{raw environment state is still exposed}.   We introduce PlanTwin, a privacy-preserving architecture for cloud-assisted planning without exposing raw local context. The key idea is to project the real environment into a \textit{planning-oriented digital twin}: a schema-constrained and de-identified abstract graph that preserves planning-relevant structure while removing reconstructable details. The cloud planner operates solely on this sanitized twin through a bounded capability interface, while a local gatekeeper enforces safety policies and cumulative disclosure budgets. We further formalize the privacy-utility trade-off as a capability granularity problem, define architectural privacy goals using $(k,δ)$-anonymity and $ε$-unlinkability, and mitigate compositional leakage through multi-turn disclosure control.   We implement PlanTwin as middleware between local agents and cloud planners and evaluate it on 60 agentic tasks across ten domains with four cloud planners. PlanTwin achieves full sensitive-item non-disclosure (SND = 1.0) while maintaining planning quality close to full-context systems: three of four planners achieve PQS $> 0.79$, and the full pipeline incurs less than 2.2\% utility loss.

### 1. 研究动机与核心问题
*   **研究背景**：当前智能体系统（如 Claude Code, GitHub Copilot）通常采用“云端规划、本地执行”的架构。云端大模型（LLM）作为规划器需要访问本地环境（代码、文件、凭据等）来生成计划。
*   **核心问题**：现有的安全机制（如沙箱隔离、访问控制、PII 脱敏）主要关注“智能体能做什么”，而忽略了“规划器能看到什么”。在授权范围内，原始的私有上下文仍会被发送至云端，导致严重的隐私泄露风险。
*   **研究动机**：如何在不向云端暴露原始本地数据的前提下，利用云端顶级 LLM 的强大规划能力？

### 2. 方法论与关键技术
PlanTwin 提出了一种**“拆分架构”**，将规划与原始数据解耦。
*   **核心思想**：将真实环境投影为一个“面向规划的数字孪生（Digital Twin）”——一个受模式约束、去标识化的抽象图。云端规划器仅在抽象图上操作，而所有原始数据访问和执行均留在本地。
*   **四阶段本地投影流水线 (Π)**：
    1.  **类型化提取 (Typed Extraction)**：利用本地小模型 (SLM) 或启发式算法识别对象的语义角色（如：配置文件、敏感容器）。
    2.  **敏感实体脱敏 (Redaction)**：使用正则和规则屏蔽路径、密钥、IP 等。
    3.  **值泛化 (Generalization)**：将精确值映射到粗粒度桶（如：文件大小设为“大/中/小”，时间设为“近期/陈旧”）。
    4.  **模式投影 (Schema Projection)**：输出固定的 JSON 抽象图，不含任何原始文本。
*   **能力化规划接口 (Capability Interface)**：云端仅能调用预定义的、声明式的本地能力（如 `security_audit`），而非生成任意代码。
*   **本地网关 (Gatekeeper)**：
    *   **披露预算管理**：为每个对象分配“披露预算”，根据暴露的字段权重累积计算，防止通过多轮对话“拼凑”出原始信息。
    *   **策略强制执行**：验证云端计划的合规性，必要时引入人工审批。

### 6. 主要结论与发现
*   **隐私性**：PlanTwin 实现了 **SND = 1.0**（敏感项零泄露），而传统的 PII 脱敏在代码任务中会泄露高达 88% 的秘密。
*   **效用性**：在 3/4 的云端规划器上，PlanTwin 的规划质量 (PQS > 0.79) 接近全上下文系统，整体效用损失 **< 2.2%**。
*   **发现**：强大的云端规划并不依赖原始文本，只要提供正确的结构化语义和能力接口，抽象信息足以支撑高质量规划。
*   **本地模型局限**：即使是 9B 规模的本地模型，其规划能力也远逊于云端辅助的 PlanTwin。

### 7. 优点
*   **架构级保障**：通过“数字孪生”抽象层，从根本上切断了原始数据流向云端的路径。
*   **动态预算控制**：创新的披露预算机制解决了交互式系统中隐私泄露随轮数累积的难题。
*   **低开销**：对本地算力要求极低，且由于传输的是压缩后的抽象图，有时能降低云端 API 的 Token 成本。

### 8. 不足与局限
*   **提取质量依赖**：第一阶段的类型提取如果出错（如将敏感文件误分类），可能影响后续规划。
*   **结构化泄露风险**：虽然隐藏了内容，但图的拓扑结构（如依赖关系）仍可能泄露某些宏观信息（如项目架构）。
*   **延迟问题**：引入本地 SLM 进行语义提取会显著增加首轮响应时间（约 10-30s）。
*   **应用限制**：目前主要针对结构化程度较高的任务（如代码、数据库），对于极度依赖非结构化文本细节的规划任务可能效果受限。
