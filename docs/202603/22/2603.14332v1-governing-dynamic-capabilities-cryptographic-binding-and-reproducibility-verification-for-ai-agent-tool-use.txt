## Abstract 

AI agents dynamically acquire capabilities at runtime via MCP and A2A, yet no framework detects when ca-pabilities change post-authorization. We term this the 

capability-identity gap : it enables silent capability esca-lation and violates EU AI Act traceability requirements. We propose three mechanisms. Capability-bound agent certificates extend X.509 v3 with a skills manifest hash; any tool change invalidates the certificate. Reproducibility commitments leverage LLM inference near-determinism for post-hoc replay verification. A verifiable interaction ledger provides hash-linked, signed records for multi-agent forensic reconstruction. We formalize nine security proper-ties and prove they hold under a realistic adversary model. Our Rust prototype achieves 97 μs certificate verification (<1 ns capability binding overhead, ∼1,200 ,000 × faster than BAID’s zkVM), 0.62 ms total governance overhead per tool call (0.1–1.2% of typical latency), and ∼2,230 ledger appends/s. We validate reproducibility across 9 models from 7 providers (15,120 pairwise comparisons): same-provider outputs achieve 4.7 × separation from cross-provider outputs (Cohen’s d > 1.0 on all four metrics), with best classification at F1=0.876 (Jaccard, θ=0.408); single-provider deployments achieve F1=0.990 with 11.5 ×

separation. We evaluate 12 attack scenarios—silent escala-tion, tool trojanization, phantom delegation, evidence tam-pering, collusion, and runtime behavioral attacks validated against NVIDIA’s Nemotron-AIQ traces—each detected with a traceable mechanism, while the MCP+OAuth 2.1 baseline detects none. An end-to-end evaluation over a 5-to-20-agent pipeline with real LLM calls confirms that full governance (G1–G3) adds ∼10.8 ms per pipeline run (0.12% overhead), scales sub-linearly per agent, and de-tects all five in-situ attacks with zero false positives. 

## 1 Introduction 

1.1 Agent Governance: A Missing Foun-dation 

AI agents are no longer isolated tools. Through proto-cols like MCP (Model Context Protocol) [ 8 ] and A2A (Agent-to-Agent Protocol) [ 8], modern agents dynami-cally discover tools, invoke external APIs, execute code, and orchestrate other agents at runtime. The MCP ecosystem alone has grown to over 1,000 tool servers in under a year [ 32 ]. As these agents assume real-world responsibilities—managing enterprise credentials, execut-ing financial transactions, accessing sensitive data—a fun-damental question emerges: what properties must an agent ecosystem satisfy for humans to trust it? 

This question has been raised from multiple directions. The EU AI Act (Regulation 2024/1689) mandates trace-ability, logging, and conformity assessment for high-risk AI systems (Articles 12, 43, 72). NIST’s AI Agent Standards Initiative [ 26 ] identifies security controls, governance over-sight, and monitoring as key requirements. The TRiSM framework [ 28 ] structures agent governance around ex-plainability, security, privacy, and lifecycle management. Schroeder de Witt [ 11 ] calls for secure interaction proto-cols that prevent insecure emergent behaviors. Cheng et al. [ 43 ] ground agent safety in transparency, accountabil-ity, and trustworthiness. Allegrini et al. [44] formalize 31 safety and liveness properties for agentic task lifecycles in temporal logic. Ghosh et al. [ 45 ] propose an opera-tional safety framework with red-teaming methodology, policy-based tool scoping, and optional trace signing—but without cryptographic capability binding, formal adver-sary models, or verifiable replay mechanisms. Yet despite this convergence of concern, no existing framework translates these governance aspirations into enforceable cryptographic infrastructure . The gap is not in recognizing what is needed, but in building it. We propose three Agent Governance Requirements 

that distill the common thread across these efforts into technically precise, verifiable properties: 

Definition 1 (Agent Governance Requirements) . A gov-erned agent ecosystem satisfies three properties: 

G1: Capability Integrity. Every agent’s identity is cryptographically bound to its complete capability set—the model it executes and every tool it can invoke. The identity is unforgeable and non-repudiable. Any change to the capability set invalidates existing credentials and requires explicit re-authorization by a human or organizational principal.

G2: Behavioral Verifiability. An agent’s actions within its authorized capability envelope can be indepen-dently verified post-hoc, enabling detection of model sub-stitution, unauthorized tool use, or output manipulation. 

G3: Interaction Auditability. All inter-agent and agent-human interactions produce tamper-evident records sufficient for forensic reconstruction, allowing humans to audit agent behavior, attribute responsibility, and provide evidence for regulatory compliance and system improve-ment. 

These requirements are not novel in aspiration—they reflect the consensus of legal [ 33 , 34 ], regulatory [ 26 ], and technical [11, 28, 43, 44, 45] communities. What is novel is the claim that all three must hold simultaneously , and that none is achievable with existing agent infrastructure .G1 alone (without G2 and G3) permits certified agents to misbehave undetectably within their capability envelope. G2 alone (without G1) cannot distinguish authorized from unauthorized capabilities. G3 alone (without G1 and G2) produces audit logs of unverified actions by unverified agents. 

1.2 Why the Current Landscape Falls Short 

The current agent security landscape can be mapped into four layers (Figure 1), each addressing fragments of the governance requirements but none satisfying them: 

Identity platforms (Keyfactor [ 19 ], SPIFFE/SPIRE [ 21 ,22 ]) issue certificates binding an agent to a public key— but a certificate remains valid regardless of how many tools the agent acquires after issuance. G1 violated :capability changes are invisible to the identity layer. 

Authorization frameworks (OIDC-A [ 4], Authenticated Delegation [ 3], Cerbos [ 39 ]) define what an authenticated agent may access—but authorization is evaluated against declared capabilities at grant time, with no mechanism to detect drift. G1 violated : authorization does not track capability changes. 

Runtime gateways (Gravitee [ 40 ], Lasso Security [ 41 ]) inspect tool calls in real time—but are fundamentally stateless, with no memory of prior tool configurations and no behavioral verification. G2 and G3 violated .

Governance platforms (Credo AI [ 42 ]) maintain agent registries documenting capabilities—but operate at the documentation layer without cryptographic enforcement. 

G1 violated : declared capabilities are not bound to the agent’s actual configuration. The root cause is a failure to recognize a fundamental architectural distinction in LLM-based agents. Inside a transformer’s forward pass, tool definitions and user con-text are indistinguishable token sequences. But at the orchestration layer, tool definitions determine which real-world actions the model’s outputs can trigger ,while runtime context determines which actions the model chooses . 1.3 Our Framework: Bridging the Gap 

The capability-context separation provides the theoretical foundation for three mechanisms, each targeting a specific governance requirement: 

Capability-Bound Agent Certificates (§3) → G1. 

We extend X.509 v3 with a skills manifest hash —a SHA-256 commitment to the agent’s complete tool configuration. Any mismatch causes immediate certificate invalidation. Certificates are embedded within a trust propagation tree 

where human and organizational entities form the roots, and trust constraints (maximum credential tier, delegation depth, allowed tools, rate limits) propagate monotonically downward. No agent can exceed the trust boundaries set by its human principal. 

Reproducibility Commitments (§4) → G2. Given that the capability envelope is fixed (certificate-bound), we leverage LLM inference near-determinism [ 1, 6 ] to enable post-hoc replay verification. Agents declare a reproducibil-ity commitment level in their certificate. When an incident occurs, a verifier re-executes the agent’s declared model 2with the recorded inputs and compares outputs. Diver-gence beyond the committed threshold proves the agent executed something other than what its certificate claims. 

Verifiable Interaction Ledger (§5) → G3. Each interaction record contains agent IDs, certificate hashes, content commitments, and reproducibility anchors, linked by a hash chain with per-record signatures. The ledger stores only cryptographic commitments (not raw content), preserving privacy while enabling forensic reconstruction of multi-agent chains. Together, these three mechanisms provide what the existing landscape lacks: a stateful, cryptographic gov-ernance layer satisfying G1–G3, bridging identity plat-forms and runtime gateways. Our contribution is not in novel cryptographic primitives; rather, we show that 

standard mechanisms —X.509 certificates, SHA-256 hash-ing, Ed25519 signatures, hash chains—when composed through the capability-context separation, are sufficient to enforce G1–G3 with negligible overhead, providing ac-tionable design guidance for how agents should be built, how they should interact, and how their behavior should be audited.

The adversary cannot compromise non-agent nodes or break standard cryptographic primitives. Under this model, we define and prove nine security properties: 1. ## 2 Threat Model and Motivating Attacks 

2.1 System Model 

We model the trust infrastructure as a rooted directed tree T = ( V, E, r, τ ), where V is the set of all entities (CAs, humans, organizations, AI agents), E is the set of certificate issuance relationships, r is the root CA, and 

τ : V → { NA , AG } classifies each node as Non-Agent (NA ) or Agent ( AG ). The tree satisfies a structural type constraint : τ (u) = 

AG =⇒ ∀(u, v ) ∈ E : τ (v) = AG . Agents can delegate to sub-agents, but cannot confer trust upon humans or legal entities. Each agent node v holds a Capability-Bound Certificate 

cert v and maintains a runtime tool configuration Tv (t)

that may change over time. Each credential c has a risk tier tier c ∈ { T 0, T 1, T 2, T 3} where T 0 (human-only) is most sensitive. 

2.2 Adversary Model 

We define a PPT adversary A with capabilities: C1: Network Control (Dolev-Yao). C2: Agent Com-promise of up to k agent nodes. C3: Rogue Agent Creation with fresh key pairs. C4: Silent Capabil-ity Escalation —modifying tool configurations without changing identity credentials. C5: Interaction Injec-tion —fabricating or modifying ledger records. 

A cannot : compromise non-agent nodes, forge digital signatures (EUF-CMA), find hash collisions, or reverse hash functions. 

2.3 Motivating Attack Scenarios 

We enumerate twelve attacks organized into four cate-gories. ATK-1 through ATK-9 address structural gover-nance threats; ATK-10 through ATK-12 address runtime behavioral attacks empirically validated by the Nemotron-AIQ dataset [ 45 ], which demonstrates 100% success rates for prompt injection and tool-mediated attacks against unprotected agentic workflows. 

Capability Attacks. ATK-1: Silent Capability Es-calation. Agent v certified with {code_gen , unit_test }

acquires web_browser via a plugin marketplace. Under every existing framework, v’s credentials remain valid. Our framework detects the skills manifest hash change. 

ATK-2: Model Substitution. A replaces the certified model with a fine-tuned variant. Detected via model binding mismatch and replay verification divergence. 

ATK-3: Tool Trojanization. A replaces tool implemen-tation (same name/version, different code). Detected via code-level hashing in the skills manifest. 

Delegation Attacks. ATK-4: Phantom Delegation. 

Compromised agent (T2) issues T1 certificate to sub-agent. Prevented by monotonic trust propagation. ATK-5: Chain Forgery. Fabricated certificate chain. Prevented by signa-ture verification at every level. ATK-6: Depth Overflow. 

Deep delegation hierarchy. Prevented by strictly decreas-ing depth counters. ATK-7: Credential Collusion. Two or more compromised agents at lower tiers (e.g., T3 +T3) attempt to combine their credentials to access resources restricted to a higher tier (T1). Prevented by indepen-dent per-agent credential evaluation—each agent’s tier and permissions are verified in isolation, so colluding agents gain no additional permissions beyond their individual certificates (Theorem 10). 

Forensic Attacks. ATK-8: Evidence Tampering. Log modification after incident. Prevented by hash-linked ledger with bilateral signing. ATK-9: Blame Shifting. 

Dispute in multi-agent chain. Resolved via interaction commitments and reproducibility replay. 

Runtime Behavioral Attacks. ATK-10: Injection-Driven Knowledge Poisoning. Adversarial content in-jected via tool outputs (e.g., retrieval results) propagates through the agent’s reasoning chain, corrupting final out-puts. Nemotron-AIQ traces show this attack succeeds in 100% of undefended trials across knowledge-poisoning and phishing-link templates. Our ledger’s content com-mitments ( ηin , η out ) enable post-hoc identification of the exact span where poisoned content entered the chain. 

ATK-11: Cascading Action Hijack. Injected instruc-tions persist across multiple agent workflow stages (IN-FECT template), causing the agent to refuse legitimate tasks or produce harmful content at downstream nodes. Our hash-chain ledger records every inter-span transition, enabling forensic tracing of attack propagation paths that OTEL-only audit trails cannot reliably provide. 

ATK-12: Output Exfiltration via Rendering. Agent outputs are manipulated to embed phishing links (PHISH-ING_LINK) or malicious markdown (MARKDOWN) that exfiltrate data when rendered by downstream consumers. Detected post-hoc via reproducibility verification: replay-ing the interaction with the committed parameters pro-duces benign output, diverging from the tampered record.

## 4 Reproducibility Commitments 

4.1 Reproducibility as a Necessary Con-dition 

A key observation motivates our approach: behavioral verifiability (G2) is impossible without repro-ducibility . If an agent’s outputs cannot be indepen-dently regenerated—even approximately—then no post-hoc mechanism can distinguish authorized behavior from unauthorized behavior. This is not merely a limitation of our framework; it is a fundamental constraint on any 

verification scheme that does not rely on trusted execution environments. This observation has a prescriptive implication for se-cure agent system design: high-assurance agent de-ployments must be engineered for reproducible output. Concretely, this means using deterministic sam-pling (temperature =0, fixed seeds), preferring structured outputs (code, JSON, SQL) over free-form prose where security-critical decisions are involved, and selecting in-ference providers that guarantee bitwise or near-bitwise reproducibility [ 6]. Agents that cannot commit to any reproducibility level receive a trust tier downgrade (§4.4), making the security cost of non-reproducibility explicit in the governance framework. 

4.2 Empirical Foundation 

LLM inference is nearly deterministic . Karvonen et al. [ 1]demonstrated that with fixed sampling parameters, regen-erating output produces >98% token-identical sequences. 5EigenAI [ 6 ] achieved 100% bit-exact deterministic infer-ence on fixed GPU architectures with negligible perfor-mance loss. Our own validation (§7, Experiment 5) con-firms that code-generating agents achieve perfect determin-ism, while prose outputs exhibit paraphrase-level variation that remains well-separated from cross-model divergence.

4.8 Chain Verifiability 

The preceding analysis establishes that G2 can verify an individual agent’s computational process. A natural question arises: when agents form a multi-hop execution chain, is per-node verification sufficient for end-to-end behavioral verifiability? We show that it is not—behavioral verifiability is a chain property , not a node property— and that this observation has direct implications for trust tree architecture (Figure 4). 

The gap that G2 uniquely fills. Consider what G1+G3 alone guarantee. G1 (capability binding) ensures that each agent’s certificate is bound to its declared model and tool set; any capability change invalidates the certifi-cate. G3 (interaction auditability) ensures that every input and output is bilaterally signed and hash-chain linked; no record can be silently modified. Together, G1+G3 provide 

attribution integrity : we know who sent what to whom ,and that no message was altered in transit. However, signing proves attribution, not computa-tional fidelity . A compromised agent holding a valid cer-tificate (G1 ✓) can execute an arbitrary model internally, produce output, and sign it (G3 ✓). The ledger faithfully records the interaction. Every cryptographic check passes. The only mechanism that detects this attack is G2: replay-ing the agent’s declared model on the recorded input and comparing outputs. G2 is the software-only equiva-lent of trusted execution —it verifies that the declared computational process actually produced the observed output, without requiring hardware attestation (TEE). 

Why chain position matters. Now consider a trust tree execution chain v0 → v1 → · · · → vn, where v0 is a human principal (trust anchor) and each vi (i ≥ 1) is an agent node. Agent vj receives input from vj−1, processes it using its declared model, and passes output to vj+1 .Suppose vj is a compromised agent with G2 =none 

(either because it uses a low-determinism model or because G2 verification is not enforced). Then: 1. vj ’s output cannot be verified as originating from its declared model. 2. vj ’s output becomes vj+1 ’s input. Even if vj+1 satis-fies G2 =full , replay verification of vj+1 only proves that vj+1 faithfully processed the input it received— but that input itself is unverifiable. 3. By induction, all downstream nodes vj+1 , . . . , v n op-erate on data whose provenance cannot be crypto-graphically verified back to the trust anchor.

Architectural implications. This theorem yields three deployment principles: 

Principle 1: High-determinism models at interior nodes. 

Models with strong reproducibility (Anthropic Haiku: CharMatch =0.987, Sonnet: 0.823; see Table 4) should be placed at coordinator and interior positions in the trust tree, where their outputs become inputs for down-stream agents. Low-determinism models (Kimi: 0.169, DeepSeek: 0.225) should be constrained to leaf positions where their outputs do not propagate further. This is not merely a performance recommendation—it is a security architecture constraint .

Principle 2: Verification boundary checkpoints. When a G2 =none agent must appear at an interior position (e.g., a reasoning model as coordinator), the chain requires a verification boundary : either (a) a human-in-the-loop checkpoint that re-anchors trust by reviewing and sign-ing the output, (b) TEE attestation providing hardware-enforced computational integrity, or (c) dual-execution with independent instances and output comparison (anal-ogous to Byzantine fault tolerance). Each mechanism re-establishes chain verifiability for downstream nodes. 

Principle 3: Explicit degradation in the ledger. When chain verifiability is broken at node vk, the ledger should 7record a PARTIAL_VERIFIABILITY marker at that bound-ary. Downstream agents inherit a trust tier penalty be-yond the standard G2 =none downgrade (§4.4), making the reduced security guarantee visible to auditors and compliance systems. 

Relationship to TEE. This analysis clarifies the com-plementary roles of G2 and trusted execution environ-ments. G2 provides software-only verification of computa-tional fidelity—a “software TEE”—using only black-box API access and the near-determinism of LLM inference. Hardware TEEs provide the same guarantee through hard-ware attestation but require infrastructure unavailable in most LLM deployments. The Chain Verifiability Theorem shows precisely where each mechanism is needed: at every interior agent node in the trust tree. In mixed deploy-ments, G2 and TEE are interchangeable at the chain level: an agent with TEE attestation and G2 =none maintains chain verifiability identically to an agent with G2 =full 

and no TEE. 

## 5 Verifiable Interaction Ledger 

5.1 Motivation 

Capability binding and reproducibility verify individual agents. Real-world incidents involve chains of interactions. We need an immutable, tamper-evident record where no single party—including the ledger operator—can unde-tectably alter history. 

Why OpenTelemetry traces are insufficient. The industry standard for agentic observability is OpenTeleme-try (OTEL) [ 46 ], whose span trees record execution traces with span_id /parent_id linkage, timestamped 

input.value /output.value payloads, and span-kind an-notations ( CHAIN , LLM , TOOL ). Analysis of the Nemotron-AIQ Agentic Safety Dataset [ 45 ]—10,796 OTEL traces from NVIDIA’s AI-Q research assistant under 22 distinct attack templates—reveals three critical gaps: (1) No in-tegrity protection : span linkage is purely referential; any party with storage access can modify, delete, or reorder spans undetectably. (2) No identity binding : spans record function names but not cryptographically verified agent identities or capability certificates. (3) No privacy preser-vation : OTEL stores full plaintext input/output payloads, creating data exposure risk. Our interaction ledger ad-dresses all three gaps through hash-chain linking ( hprev ), Ed25519 bilateral signing ( sig i), agent certificate binding (ch send , ch recv ), and commitment-only storage ( ηin , η out ).

5.3 Privacy Preservation 

The ledger stores only commitments (hashes), not content. During normal operation, only interaction metadata is revealed. During forensic investigation, parties produce original content verified against stored commitments. 

5.4 Forensic Reconstruction Protocol 

The protocol proceeds in five steps: (1) identify the rele-vant interaction chain, (2) verify ledger integrity (signa-tures + hash chain), (3) request content disclosure and verify against commitments, (4) verify agent certificates at interaction time, and (5) execute replay verification for agents with reproducibility commitments.

## 8 Limitations and Future Work 

8.1 Scope of Contribution 

Our contribution is a governance framework , not novel cryptographic primitives. The individual building blocks— X.509 certificates, SHA-256, Ed25519, hash chains—are standard. The novelty lies in (a) identifying the capability-context separation as the principled boundary for agent governance, (b) showing that composing standard mecha-nisms along this boundary enforces G1–G3 with negligible overhead, and (c) demonstrating this is practically achiev-able. We view this as analogous to Certificate Trans-parency [ 16 ], which composes Merkle trees and gossip protocols—neither novel individually—into infrastructure that fundamentally changed how the web PKI operates. The security proofs are intentionally straightforward re-ductions to standard assumptions (collision resistance, EUF-CMA); the contribution is in what to build , not in 

new ways to prove things .

8.2 Capability Binding Limitations 

Capability-context boundary. Our sharp boundary has edge cases: a “code execution” tool can effectively acquire new capabilities through composition (e.g., down-loading and running arbitrary scripts). Our framework guarantees that the registered capability set has not changed, but the effective attack surface within a fixed capability envelope can still be large. Complementary approaches (sandboxing, taint tracking, output filtering) are needed to govern behavior within the envelope. We view capability binding and intra-capability governance as orthogonal layers: the former ensures the attack surface is known; the latter constrains behavior within it. 

Closed-source binding. For closed-source models and tools, the skills hash captures configuration metadata (name, version, API schema), not implementation code. A provider could change implementation without changing the public descriptor. Reproducibility verification provides a complementary detection mechanism: implementation changes that alter behavior will be detected via replay divergence. 

Re-signing frequency. Mitigated through wildcard tool categories, short-lived auto-renewable certificates, and the empirical observation that tool configurations change far less frequently than user context. 

8.3 Reproducibility Limitations and Graceful Degradation 

Reproducibility is a design constraint, not a uni-versal guarantee. Our framework makes explicit what is often implicit: non-reproducible agent behavior is non-verifiable . This is not a limitation unique to our approach— it applies to any verification scheme not based on trusted hardware. For high-assurance deployments, this means agents should be configured for deterministic output (struc-tured formats, temperature =0, fixed seeds). Our empirical results (§7) show that code-generating agents already sat-isfy this constraint perfectly, and the separation between same-model and cross-model distributions ( 11 .5×) pro-vides robust detection even for prose tasks. 

Graceful degradation under partial G2. Our multi-provider experiment (§7, Experiment 5c) reveals that model determinism varies by over an order of magnitude: from CharMatch =0.987 (Haiku 4.5) to 0.169 (Kimi K2.5). This raises a practical question: what security guarantees hold when some agents use low-determinism models? When G2 is unavailable (reproducibility commitment =

none ), the framework degrades to G1+G3 only. Table 12 characterizes the precise security impact. Table 12: Security guarantees under G1+G2+G3 vs. G1+G3 degradation.

The three attacks that require G2—model substitution (ATK-2), cascading injection via model compromise (ATK-10/11 when the injection occurs through model replacement rather than tool replacement), and output exfiltration via model tampering (ATK-12)—share a common structure: the attacker operates within a valid capability envelope by substituting the computational process itself. Signing proves attribution; only G2 proves computational fidelity. 

Chain-level degradation. The Chain Verifiability Theorem (§4.8) establishes a stronger result: a G2 =none 

agent at an interior position breaks behavioral verifica-tion for all downstream nodes. This means the true cost of deploying a low-determinism model at a coordinator position is not merely that the coordinator itself is un-verifiable, but that the entire sub-tree it governs loses G2 protection. The framework’s three commitment levels (full /statistical /none ) are thus not equivalent—their 14 security implications depend critically on chain position. 

Statistical vs. computational security. Token-level reproducibility provides statistical rather than computa-tional security. However, the Bounded Divergence Theo-rem (§4.7) shows this gap is operationally small: sufficient replay verification transforms statistical detection into a provable safety certificate. An adversary who fine-tunes a model to pass n diverse replay checks has produced a model that is (n, ϵ )-indistinguishable from the original, with ϵ decreasing exponentially in n. Future work will ex-plore semantic similarity (embedding cosine distance) as a verification metric, which may provide tighter separation bounds and is robust to paraphrase-level variation. 

Provider cooperation. Replay verification requires access to the same model version. Organizations should ne-gotiate replay rights as part of service agreements. Deter-ministic inference providers [ 6] eliminate this dependency entirely. 

8.4 Trust Infrastructure Limitations 

Non-agent node compromise. We assume non-agent nodes (CAs, humans, organizations) are uncompromised. Standard PKI mitigations apply: Certificate Transparency logs, hardware trust anchors (TPM 2.0), and multi-party issuance protocols. Ledger operator trust. The current single-operator ledger relies on bilateral signing for tamper evidence. A multi-operator design with consensus is possi-ble but introduces latency overhead. Application-layer data flow. Our framework governs agent identity and capability , not the content of data flows. Complementary approaches (data-flow taint tracking, differential privacy) address this orthogonal concern. 

8.5 Future Work 

Hardware-attested agent identity (TPM 2.0, TEEs) for stronger G2 guarantees without provider cooperation. Embedding-based semantic similarity for reproducibility verification with formal threshold analysis. Cross-protocol capability governance beyond MCP/A2A. AI-assisted cer-tificate issuance to reduce operational burden. Dynamic trust calibration from ledger history, enabling trust scores to evolve based on observed agent behavior. 

## 9 Related Work 

We organize related work along the four-layer landscape. 

Identity Platforms. Keyfactor [19 ] and HID Global [ 20 ] propose X.509 for agents but without tool binding. SPIFFE/SPIRE [ 21 , 22 ] provides workload iden-tity. CyberArk [37] extends PAM to agents. 

Authorization Frameworks. OIDC-A [ 4] extends OpenID Connect with agent claims. South et al. [ 3 ] pro-pose Agent-ID Tokens (ICML 2025). Agentic JWT [ 10 ]defines delegation with temporal constraints. 

Runtime Gateways. Gravitee [ 40 ] and Lasso [ 41 ]provide MCP proxies with per-call enforcement— fundamentally stateless. 

Governance Platforms. Credo AI [ 42 ] maintains agent registries at the documentation layer. 

Verifiable Execution. BAID [ 2] provides zkVM-based verification ( ∼120 s per proof). DiFR [ 1] demonstrates inference near-determinism. EigenAI [ 6 ] achieves 100% deterministic inference. Attestable Audits [7] uses TEEs. 

Classical Trust Management. PolicyMaker [ 12 ], KeyNote [13], SPKI/SDSI [14], F-PKI [16], ARPKI [17]. 

Agent Protocols. A2A [ 8] provides agent communica-tion. MCP enables agent-tool interaction with OAuth 2.1. 

Agent Governance Frameworks. Cheng et al. [ 43 ]propose a three-pillar model (transparency, accountability, trustworthiness) as a conceptual governance framework. Allegrini et al. [ 44 ] formalize 31 task-lifecycle properties in CTL/LTL but address operational correctness rather than cryptographic governance. Ghosh et al. [ 45 ] pro-pose an operational safety framework with agent red-teaming probes (ARP), policy-based tool scoping, and OpenTelemetry-based audit trails, validated through a 10,796-trace dataset (Nemotron-AIQ) covering 22 secu-rity attack templates and 13 content-safety categories against NVIDIA’s AI-Q research assistant. Their empiri-cal results demonstrate that without guardrails, all 2,596 security attacks succeed (100%), and 20% of 2,800 safety attacks propagate to the final output—providing com-pelling evidence for the inadequacy of relying solely on model alignment. However, their OTEL-based audit trails lack cryptographic integrity: span linkage is purely refer-ential (no hash chaining), spans carry no agent identity binding (no certificates), and full plaintext I/O is stored (no privacy preservation). Our framework addresses these gaps with a formally verified, hash-chain-linked, bilaterally signed interaction ledger (Section 5). Furthermore, their “signed schemas” and “per-call capability tokens” remain design principles without cryptographic specification or formal proofs. Raza et al. [ 28 ] adapt TRiSM for agentic multi-agent systems. 

Regulatory Landscape. The EU AI Act (Regulation 2024/1689) requires traceability but lacks agent-specific implementing acts [ 33 ]. NIST CAISI [ 26 ] is developing agent standards. CSA [ 24 , 25 ] proposes agentic trust frameworks. 

## 10 Conclusion 

AI agents increasingly acquire capabilities at runtime, yet no existing framework enforces the three governance requirements—capability integrity, behavioral verifiability, and interaction auditability—that a trustworthy agent ecosystem demands. We addressed this through the capability-context sepa-ration principle and three mechanisms: capability-bound certificates (G1), reproducibility commitments (G2), and a verifiable interaction ledger (G3). We formalized nine security properties (including chain verifiability: behav-ioral verification is a chain property broken by any un-verifiable interior agent), implemented the framework in 15 Rust with 97 μs verification and <1 ns capability binding overhead ( ∼1,200 ,000 × faster than BAID), validated re-producibility across 9 models from 7 providers ( F1 = 0 .876 

multi-provider, F1 = 0 .990 single-provider), and verified 12 attack scenarios—including runtime behavioral attacks validated against NVIDIA Nemotron-AIQ traces—with specific, traceable detection mechanisms. An end-to-end evaluation deploying the full governance stack across a 5-to-20-agent pipeline with real LLM calls confirms 0.12% overhead, sub-linear per-agent scaling, and detection of all in-situ attacks with zero false positives—while three base-lines (no governance, OAuth 2.1, OpenTelemetry) detect none. Our framework fills the stateful, capability-aware gover-nance gap between identity platforms and runtime gate-ways. As the EU AI Act’s high-risk provisions take effect and NIST develops agent-specific standards, the need for cryptographic infrastructure enforcing G1–G3 will only grow. We release the core mechanisms as an open-source SDK (agent-trust-kit ) to enable ecosystem adoption.
